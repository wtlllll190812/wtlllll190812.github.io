Unity DOTS Entities包文档翻译及学习笔记（二）常见概念解释

Mar 08, 2025
阅读时长: 6 分钟

ECS介绍

Entities包使用ECS架构组织代码和数据，实体是一个独一无二的标识符，功能上类似于一个轻量级的GameObject代替。实体表现为一个与组件相关联的ID,组件中包含了与实体相关的数据。

与GamesObjects不同，entities不包含代码，而是你所创建系统处理的最小单位

下图描述了实体，组件和系统如何共同工作

图中系统读取了速度和方向组件，将其相乘然后更新相关的坐标组件

实体A和B有渲染组件，但实体C却没有，这并不影响系统的行为，系统不需要知道渲染组件

你可以创建一个需求渲染组件的系统，这样系统将会忽略实体C,你也可以创建一个排除了渲染组件的系统，这样实体A和B将会被忽略

图中也体现出了原型（archetype）的概念，原型就是指独一无二的组件组合

实体

实体代表分散在你程序中哪些拥有自己数据的东西，类似角色，特效，UI，或者一个更加抽象的东西，类似网络传输。

实体类似于非托管的轻量级GameObject，代表了程序中一些特定的元素，但是实体表现为一个与不同组件数据相关联的ID,不包含任何代码或者作为关联组件的容器。

实体的集合存储在世界中（World）,世界的实体管理器（EntityManager）管理世界中的所有实体。

方法

实体管理器包含了一些方法，这些方法可以用来创建删除或者修改世界中的实体，下面列出了一些常见的方法：

方法 描述
CreateEntity 创建一个新的实体
Instantiate 复制一个已存在的实体并根据复制，创建一个新的实体
DestroyEntity 销毁存在的实体
AddComponent 向已经存在的实体添加组件
RemoveComponent 移除已存在实体的组件
GetComponent 获取组件的数据
SetComponnet 覆写组件数据的值

注意，创建或者销毁实体会导致结构变更（structural change）,这会影响游戏性能，这在结构变更部分会有更多介绍

实体没有类型，但你可以根据其关联的组件分类实体，试题管理器会持续追踪现存实体的每种独特组件组合（原型）。

组件

在ECS架构中，组件包含了系统所要读写的数据

Unity 使用没有方法的 IComponent 接口用来将一个结构体标识为组件类型，组件类型只能包含非托管类型的数据。但如果你想创建一个托管的组件，你可以将其定义为类。组件的定义中可以包含方法，但最好保持为纯数据，

独一无二的组件组合称为原型（archetype）,ECS架构通过16KiB大小的内存块来通过原型存储数据，这些内存块称为“chunk”

组件的类型

组件类型 原名 介绍
非托管组件 Unmanaged components 最常见的组件类型，只能存储特定类型的数据成员。
托管组件 Managed Components 一种托管的组件类型，可以存储任意类型的数据成员。
共享组件 Shared components 根据实体的共享组件值将其分组到不同的块中。
清理组件 Cleanup components 当销毁一个包含清理组件的实体时，Unity 会移除该实体上所有非清理组件。这在标记需要在销毁时被清除的组件时非常有用。
标签组件 Tag components 一种非托管的组件，不存储任何数据，可以在查询中用来筛选实体。
缓存组件 Buffer components 类似于可动态调整大小的数组。
块组件 Chunk components 存储与整个块相关联的数据，而不是单个实体。
可使能组件 Enableable components 一种可以在运行时启用或禁用的组件，且不会导致昂贵的结构变更。
单例组件 Singleton components 在某个世界中唯一实例的组件。

系统

系统提供了将组件数据从上一个状态转换到下一个状态的逻辑，例如，一个系统根据所有移动中的实体的速度与距离上次更新经过的时间来更新位置。

系统每帧在主线程上运行一次，它被组织成系统组（System group）的层级结构，这样你就可以控制系统的执行次序

你既可以创建非托管的系统，也可以创建非托管的系统。 要创建托管的系统，你需要定义一个继承自“SystemBase”的类。 要创建非托管的系统，你需要定义一个继承自ISystem的结构体

无论是ISystem还是SystemBase，你都有三个方法可以重写，分别是：OnUpdate,OnCreate以及OnDestory,系统的OnUpadte方法每帧执行一次

一个系统只能在一个世界中处理实体，所以系统会被关联到某个特定的系统，你可以使用World属性来返回系统所属的世界。

默认地，一个自动的引导程序会创建每一个系统和系统组的实例。引导程序会创建一个默认世界，包含以下三个系统组：InitializationSystemGroup, SimulationSystemGroup, 和 PresentationSystemGroup。

用户创建的系统会默认添加到SimulationSystemGroup中，你可以使用 [UpdateInGroup] 属性来指定所在的系统组

要禁止自动的引导程序，你可以使用以下宏定义#UNITY_DISABLE_AUTOMATIC_SYSTEM_BOOTSTRAP

系统类型

以下列出了你可以使用的系统类型

  • SystemBase：托管系统的积累
  • ISystem：非托管系统的接口
  • EntityCommandBufferSystem：为其他系统提供实体命令缓存实例，这可以将结构变更聚集在一起提升游戏性能
  • ComponentSystemGroup：用于定义系统组，为系统提供嵌套以及定义更新顺序的功能

系统组

系统组可以将系统或者其他系统组作为子系统，系统组有一个可以重写的Update方法，基类方法会按照排好的顺序更新子系统

监视系统

可以使用System窗口来观察每个世界中系统的更新顺序，以及系统整体的层级结构

世界（World）

世界就是实体的集合，实体的ID在所在的世界是独一无二的。世界中有一个 EntityManager 结构体用来创建删除以及修改实体。

世界拥有一组系统，这些系统通常只会访问同一世界的实体。 同一世界中有相同组件类型组合的实体被一起存储在原型中，原型决定了程序中的组件在内存中如何组织。

初始化

磨人来说，当你进入Play模式后，unity会创建一个World实例，并将每个系统添加到默认的世界中。

如果你倾向于手动将系统添加到默认的世界，你可以创建一个实现了IcustomBootstrap的类。

如果你想完整地手动管理引导程序，可以使用这些宏定义禁用默认世界创建：

  1. #UNITY_DISABLE_AUTOMATIC_SYSTEM_BOOTSTRAP_RUNTIME_WORLD,关闭默认运行时World的生成
  2. #UNITY_DISABLE_AUTOMATIC_SYSTEM_BOOTSTRAP_EDITOR_WORLD,关闭默认编辑器World的生成
  3. #UNITY_DISABLE_AUTOMATIC_SYSTEM_BOOTSTRAP，关闭所有默认世界的生成（以上两者）

原型

原型是一种独一无二的标识符，用于标识世界中所有具有相同组件组合的实体。举个例子，在下图中，所有只具有 speed, Direction,position 和 renderer 的实体共享一个原型X;所有只拥有 speed , direction 和 Position 的组件共享一个原型Y

当你在一个实体上添加或者删除某个组件时，该世界的实体管理器将会把该实体移动到对应的原型中，例如，当一个实体有着 Speed, Direction 和 Position 组件，而你移除了其中的 Speed 组件，实体管理器就会将该实体移动到只有 Direction和 Position的原型中，如果没有这样的原型，实体管理器就会创建一个

基于原型来组织实体意味着根据组件类型查询实体将会变得非常高效，例如，如果你想要找到所有拥有组件A和组件B的实体，你可以找到所有含有这些组件的原型，这比遍历所有单独的实体要快得多，
